1. Associating Menus and Buttons Clicks to Views or Controllers

1.1 Associating clicks to a View

  • Maya framework allows you to load a view without any direct controller.
  • The utility generateView.php automatically can generates views at ../usr_generated/view and you can extend/override it at ../usr/view.
  • DO NOT EDIT the files inside ../usr_generate because if you have modifications these will be overwritten by the view you generate using generateView.php utility.
  • You are only allowed to modify the codes in ../usr/view folder.
  • When user is currently logged-in, any view file can be loaded directly without controllers but when not, only files with `.pub.` which means public can be loaded like advance.pub.htm

There are basically 2 methods load a view using menus

    1.2 Associating Menu or Button Clicks to a controller

    • Maya framework allows to associate clicks to post to controllers using mx- tags.
    • The controller may return an html view, a json_encoded success-message data or another post command to follow.
    • The util/generateController.php can automatically generate controllers for inserting, updating and deleting items in each table.
    • DO NOT EDIT the generated files inside ../usr_generate because if you have modifications these will be overwritten by the view you generate using generateView.php utility.
    • You may modify the extended controllers at ../usr/controller folder.
    • Buttons with mx- tags should be placed inside a form with CSRF as shown below or the mx-form tag should be filled to indicate which form to submit during click.
    • If there are no forms nearest or defined using mx-tags, the button will submit a simple form with just CSRF data
    • CRSF can be rendered using any of the model views (eg. user_view->renderCsrf()) located at usr/view/[table]View.php (base model at /usr_generated/view/[table]ViewBase.php
    • Without the CSRF the post will be considered invalid unless you declare the controller as GET method where in you have to protect using your own key or password method

    2. Designing Controllers

    There are 3 types of controllers, restful controllers that return json encoded array(succes,message), restful controllers that return html views, and non-restful controllers

    2.1 Restful Controllers JSON Encoded Response of Success and Message

    This type of controller is best for actions that does not require any view but just a confirmation of success. For exmample, a delete function of a certain table say users table. When you want to delete a row in the users table, you send the information needed to identity the object into the controller from a form (example users.id), then all you need to know is whether the delete was successful or not. A typical delete function is shown below:

    find(array("id"=>$id)); if ($user->getId()>0) { if ($user->delete()==1) { $message="Successfully deleted ".$id; $success=true; } else { $message="SQL Error! ".$_SESSION["error"]; $success=false; } } else { $message="Cannot find user with id=".$id; $success=false; } $response = array("message"=>$message,"success"=>$success); header("Content-Type: application/json"); echo json_encode($response); exit(); }'; echo highlight_string($code)[0];?>

    Please note that the deleteRestAction() autogenerated by util/generateController.php is slightly different as involves input validation and structured so that certain sections can easily be overriden by parts.

    The deleteRestAction() can now be associated with button with class click_action and attribute url=?command=deleleRest. Please note that the Action is taken out in the command equivalent.

    Generally the response JSON is already handled by the maya system but in case you want to control on your own the posting and response processing by making your own AJAX code, you immediately parse the return code as data.success which is true or false, and the corresponding messages as data.message.

    2.3 External Link Controllers

    These are controllers that can be accessed directly through a link. Typical purpose of these controllers are email links for confirmation, chat links, or video links within the system. Warning! These links are very insecure so you must put some form of security such as a secure key that varies per transaction, or secure keys that exist only for a certain period of time. These controllers should always end in ActionGet() such as confirmActionGet(), so that for example this is on Users controller then the link created should be ?command=confirm&object=User&key=secret_key
